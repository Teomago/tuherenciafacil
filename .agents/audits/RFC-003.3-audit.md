## Section 2 - Documents Collection and R2 Uploads

### BLOCKING B-2: The Documents upload design is technically ambiguous for Payload

**Spec says:** `Documents` has a `file` field using "built-in Payload upload configuration (R2 backed)."

**Problem:** In Payload 3, upload support is configured at the collection level with the `upload` property, not by adding a normal `file` field. Context7 Payload 3.82.0 docs confirm an upload collection needs collection-level `upload` configuration, and the S3 plugin must list every upload collection it handles. The current repo only configures S3 storage for `media` in `payload.config.ts`.

**Risk:** The executor may create an invalid field, or create a non-upload collection that does not actually store files. Alternatively, they may reuse `media`, which conflicts with `PROJECT_STATE` and `ROADMAP` saying RFC-003.3 needs a dedicated `documents` collection beyond `media`.

**Required fix for decision file:** Choose one explicit approach.

Recommended approach:

- `Documents` is a dedicated upload collection:
  - `slug: 'documents'`
  - collection-level `upload` config
  - allowed MIME types: `application/pdf`, `image/jpeg`, `image/png`
  - max file size policy, if supported directly by Payload config or enforced by validation/hook
  - no separate `file` relationship field
- Update `payload.config.ts` S3 plugin:
  - add `documents` to `s3Storage({ collections: { ... } })`
  - specify whether it uses a prefix such as `documents`
  - specify whether downloads are public or signed/private

The decision must include exact file paths and exact expected configuration changes.

### BLOCKING B-3: Document access control is under-specified for sensitive legal files

**Spec says:** Clients can read documents linked to their cases; Admin/Lawyer can read all. Clients can upload. Clients can update if status is not `verified`; only Admin/Lawyer can change `status` or `verifiedBy`.

**Problems:**

1. "Admin/Lawyer can read all" is too broad. Existing case access uses assigned lawyer semantics. A member with `role: 'abogado'` should not automatically read every client's legal documents unless the business explicitly approves global lawyer access.
2. There is no exact async ownership function for checking `document.case.responsable` or `document.case.abogadoAsignado`.
3. "Admin" is ambiguous between Payload `users` collection roles and `members.role === 'abogado'`.
4. `verifiedBy` is described as relationship to `users` or `members`, but not whether it is polymorphic (`relationTo: ['users', 'members']`) or split into separate fields.
5. Client update permission conflicts with `docs/design/COLLECTIONS.md`, which says clients do not update Documents directly and must upload a new version.

**Risk:** A naive implementation can leak documents across cases, let inactive lawyers access legal documents, or let clients modify reviewed documents in place.

**Required fix for decision file:** Provide exact access rules and helper functions:

- Define "staff" as Payload `users` collection (`admin`/`editor`) and "lawyer" as active `members.role === 'abogado'`.
- Decide whether lawyers can read all cases or only assigned cases. The safer default is assigned cases only.
- Define document read policy:
  - staff: true
  - assigned active lawyer: documents for assigned cases
  - client: documents for owned cases where `visibility === 'client'`
- Define document create policy:
  - client: only for owned case
  - assigned active lawyer/staff: for assigned/all cases
- Define document update policy:
  - client: preferably false; resubmission creates a new version
  - assigned active lawyer/staff: can update review fields
- Define field-level access for `status`, `verifiedBy`, `notes`, `visibility`, and any review fields.
- Define delete policy. Current design says only admin, and documents should be versioned rather than deleted.

### HIGH H-2: The lifecycle status names conflict across spec and design docs

**Spec says:** Document `status`: `pending`, `uploaded`, `verified`, `rejected`. Checklist success criteria also refers to a verified document.

**Design/DEC-002 says:** Checklist blocks use `status: 'approved'`; DEC-002 says the Poder must be marked `Aprobado`; `docs/design/COLLECTIONS.md` uses checklist statuses `pending`, `uploaded`, `approved`, `rejected`.

**Risk:** Phase gate code may check `verified` while checklist data stores `approved`, or vice versa. This will either block valid cases or allow invalid advancement.

**Required fix for decision file:** Freeze one status vocabulary and mapping before implementation.

Recommended:

- `Documents.status`: `uploaded`, `approved`, `rejected` or `pending`, `uploaded`, `approved`, `rejected`
- `DocumentChecklist.status`: `pending`, `uploaded`, `approved`, `rejected`
- Phase gates must check checklist status `approved`, not only linked document status.

If the team prefers `verified`, then every design reference and DEC-002 gate in the decision must explicitly map `approved` to `verified` and use one term in code.

### HIGH H-3: Private document URL behavior is not specified

**Spec says:** No new custom REST endpoints are required; generated Payload APIs handle CRUD.

**Design docs say:** Files are not served with public URLs; a custom `GET /api/documents/:id/url` endpoint generates a signed R2 URL with TTL 1 hour after permission checks.

**Risk:** Legal documents could be served via public object URLs, or the executor may rely on Payload's default upload URLs without confirming access controls. This is a sensitive-data issue.

**Required fix for decision file:** Decide the RFC-003.3 boundary explicitly:

- If signed URLs are in scope, add endpoint file path, permission rules, TTL, tests, and verification.
- If signed URLs are out of scope, require admin-only/manual verification and document that client file download is deferred. Also configure storage so documents are not accidentally public.

Do not leave this to generated CRUD behavior.

## Section 3 - DocumentChecklist Engine

### BLOCKING B-4: The DocumentChecklist schema conflicts with DEC-002 and `docs/design/COLLECTIONS.md`

**Spec says:** `DocumentChecklist` has `requiredDocumentType`, `targetEntity` group, and `fulfilledBy`.

**Design/DEC-002 requires:** `nombre`, `descripcion`, `guiaDePaola`, `categoria`, `heredero`, `asset`, `status`, `required`, `receivedPhysically`, `receivedPhysicallyAt`, `document`, `reviewNote`, `reviewedBy`, `reviewedAt`, and a special Phase 0 Poder item.

**Risk:** The spec cannot implement DEC-002's strict Poder gate. It also omits the client-facing guidance fields that make the checklist operational for Paola and the client.

**Required fix for decision file:** Replace the current checklist schema with the canonical design schema or explicitly document a narrower MVP schema that still supports:

- Phase 0 Poder item generation
- `receivedPhysically` flag that does not unlock the gate
- status transitions `pending -> uploaded -> approved/rejected`
- review note and reviewer tracking
- relationship to the uploaded Document
- exact target relationships for heirs/assets without raw freeform IDs

Avoid generic `entityId: text` unless there is a clear reason. Payload relationships should be used for `heredero` and `asset`.

### BLOCKING B-5: Checklist generation rules are missing

**Spec says:** The objective includes checklist tracking, but the only fields listed are basic relationships. No generation algorithm is specified.

**Problem:** `ROADMAP.md` says RFC-003.3 includes the checklist generation/update engine. `docs/design/COLLECTIONS.md` defines concrete generation events:

- On case creation, generate "Poder de Representacion Notarial" for every case.
- When `Case.currentPhase` changes to 2, generate causante, heir, and asset requirements.
- When an heir or asset is added after phase 2, generate corresponding checklist items.

**Risk:** The executor may create an empty checklist collection with no automation. Success criteria would pass collection registration while the workflow remains non-operational.

**Required fix for decision file:** Add exact tasks and tests for generation:

- Hook or utility name, e.g. `generateChecklistItemsForCase`.
- Exact file paths for hook/utilities.
- Exact triggers:
  - `Cases` create: one Poder item, idempotent.
  - phase 2 transition: causante + all existing heirs + all existing assets.
  - `Heirs.afterChange` on create when case phase >= 2.
  - `Assets.afterChange` on create when case phase >= 2.
- Idempotency rule:
  - no duplicate checklist item for same case + category + type + target entity.
- Tests:
  - Poder item is created at case creation.
  - phase 2 generation creates expected checklist rows.
  - re-running generation does not duplicate rows.
  - `receivedPhysically: true` still does not satisfy phase 3 -> 4.

### HIGH H-4: Document -> Checklist synchronization is not specified

**Spec says:** `fulfilledBy` relationship to documents exists.

**Problem:** It does not specify how the relationship is set or how status changes. The design doc expects `Documents.afterCreate` to update the checklist item to uploaded, and lawyer review to update approved/rejected status.

**Risk:** A document can exist without satisfying a checklist item, or a checklist item can point to a document that was later rejected.

**Required fix for decision file:** Add exact lifecycle rules:

- On document create with checklist item:
  - set checklist `document` or `fulfilledBy`
  - set checklist status to `uploaded`
  - preserve previous rejected document as version history if applicable
- On lawyer approval:
  - set document status and checklist status together
  - set `reviewedBy`/`reviewedAt`
- On rejection:
  - set checklist status `rejected`
  - require `reviewNote`
  - do not delete previous upload

## Section 4 - Phase Gates and Case Advancement

### BLOCKING B-6: Phase transition rules are too vague and partially wrong

**Spec says:** A `Cases.beforeChange` hook blocks transition to Notary phase if `acuerdoHerederos` is not confirmed or checklist items lack verified documents. It also blocks transition to Payment phase if prerequisite data is missing.

**Problems:**

1. "Notary phase" is not mapped to a phase number. Current `Cases.currentPhase` is numeric `0..8`.
2. "Payment phase" is not a canonical phase in `docs/design/COLLECTIONS.md`; payments happen around case activation and later fee milestones, not as a single currentPhase value.
3. DEC-002 requires a specific Phase 3 -> 4 Poder gate.
4. DEC-004 requires tier-specific behavior for Phase 7 -> 8 and completion.
5. The spec does not prohibit skipped phases, backwards phase changes, or invalid non-integer phases.
6. The spec does not say whether the gate lives only in a `beforeChange` hook or in a custom endpoint. Design docs mention `POST /api/cases/:id/advance-phase`.

**Risk:** The executor will invent phase semantics. This is a legal workflow integrity problem, not just a code style issue.

**Required fix for decision file:** Provide the exact phase matrix:

- Phase labels for 0 through 8.
- Allowed transitions:
  - normally only `n -> n + 1`
  - define whether admin can correct backwards or skipped values
- Gate conditions:
  - 2 -> 3: no required checklist item pending/uploaded/rejected
  - 3 -> 4: all required items approved and exactly one Poder item approved; `receivedPhysically` is not enough
  - 4 -> 5: notary response approved
  - 5 -> 6: edict delivery/comprobante condition
  - 6 -> 7: DIAN and UGPP approved, if those fields are in scope
  - 7 -> 8: tier conditional from DEC-004
  - 8 -> completed: registration completed for Premium/Elite
- Error messages in Spanish.
- Test cases for each blocking transition.

Also decide whether RFC-003.3 implements a custom endpoint now. If the answer is no, the decision must explain why `beforeChange` alone is sufficient for admin verification.

---

### HIGH H-5: Case access update rules may allow unauthorized phase changes

**Spec says:** Admin/Lawyer can update phase-related data.

**Current code:** `Cases.update` allows any active assigned lawyer to update the case. There is no field-level access on `currentPhase`.

**Risk:** If RFC-003.3 only adds a `beforeChange` validation but no field-level access or endpoint boundary, any assigned lawyer update can attempt phase movement. That may be acceptable, but it must be deliberate and tested.

**Required fix for decision file:** Specify one of these:

- Field-level access for `currentPhase` and phase-related fields, plus `beforeChange` gate.
- A custom controlled endpoint for advancement, and direct updates to `currentPhase` are blocked except admin/system.

The decision must not leave both paths open without tests.

## Section 5 - NotaryProcess

### HIGH H-6: NotaryProcess is too shallow to support the phase gates described elsewhere

**Spec says:** `NotaryProcess` has `status`: `drafting`, `submitted`, `observations`, `approved`, `signed`, plus simple radicado/final deed fields.

**Design docs require:** Structured tracking for radicacion, respuestaNotario, edictos, DIAN, UGPP, firma, costosNotariales, and registro.

**Risk:** The phase gate implementation cannot check phases 4 through 8 without fields that do not exist. The executor will either skip those gates or add fields not specified in the spec.

**Required fix for decision file:** Decide whether RFC-003.3 implements the full NotaryProcess schema or a smaller MVP.

If full schema:

- Include the grouped fields from `docs/design/COLLECTIONS.md`.
- Include phase-specific tests.

If MVP:

- Scope phase gates to only the fields actually present.
- Explicitly defer phases that cannot be validated.
- Do not claim completion of "phase engine hooks" beyond the implemented gates.

**Design says:** NotaryProcess is lawyer/admin operational data; client-facing information is reflected through timeline/current phase.

## Section 6 - Payments and Wompi Boundary

### BLOCKING B-7: Payments schema does not implement DEC-004 and leaves DEC-005 boundary ambiguous

**Spec says:** `Payments` has `case`, `amount`, `currency`, `status`, `wompiTransactionId`, and `paymentReference`.

**DEC-004 requires:** Payment model must support service payments, consultation credits, tier installment logic, investigation charges, visible-vs-internal costs, and future Elite/custom charges. DEC-005 is still pending for final Wompi integration.

**Problems:**

1. No `tipo` field, so it cannot distinguish `consulta`, `servicio`, `creditoConsulta`, `investigacion`, `gasto_tercero`, etc.
2. No `member` relationship for pre-case consultation payments.
3. `case` is required, which conflicts with DEC-004 consultation credits before a case exists.
4. No `visibleParaCliente`, so internal operational costs may leak or all costs may be hidden.
5. No installment/parent payment fields, despite DEC-004 requiring package installments and future Elite compatibility.
6. The Wompi integration boundary is unclear: status fields exist, but no webhook, checksum validation, retry/idempotency policy, or explicit deferral.

**Risk:** The executor may build a payment table that immediately needs migration/rework when Wompi or package payments are implemented. Worse, it may misrepresent client-facing costs.

**Required fix for decision file:** Define the Payments foundation precisely:

- `case` optional, because consultations can occur before a case.
- `member` required or conditionally required when `case` is null.
- `tipo` enum aligned with DEC-004.
- `monto` or `amount`: choose one naming style and stick to repo language conventions.
- `metodo`/payment method enum.
- `status` and `wompiStatus` boundaries.
- `visibleParaCliente` default and rules.
- credit fields needed for `creditoConsulta`.
- Wompi in/out of scope:
  - If only foundation: no webhook, no redirect, no payment initiation, no checksum validation in RFC-003.3.
  - If webhook is in scope: include endpoint path, signature validation, idempotency, tests, and failure states.

### HIGH H-8: Payment access rules allow mutation by lawyers without immutable ledger policy

**Spec says:** Payments create/update/delete can be Admin/Lawyer or system hooks.

**Design says:** Lawyers can create offline records; updates/deletes are admin-only because payments are a ledger.

**Risk:** Payment records can be modified after the fact without audit trail. In a legal/payment context, this is a data integrity issue.

**Required fix for decision file:** Define ledger policy:

- create: staff/admin, assigned active lawyer, or system only
- update: admin/system only, or append-only corrections via a new Payment record
- delete: admin only, preferably disabled except admin
- client read: only if payment belongs to their case/member and `visibleParaCliente === true`

Add tests for client visibility and lawyer mutation restrictions.
