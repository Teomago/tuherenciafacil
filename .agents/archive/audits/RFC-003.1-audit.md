## Section 1 — Maintenance Update

### ⛔ B-1: TypeScript 6.0.2 is a major-version bump with unknown compatibility

**Spec says:** `typescript: 5.7.3 → 6.0.2`

TypeScript 6.0 is a major version. Major TypeScript releases routinely introduce breaking changes to type-checking behavior and compiler APIs. The project has zero test coverage for TypeScript type correctness. The following packages in `package.json` have peer-dependency constraints that reference TypeScript 5.x and have not been verified against TypeScript 6:

- `eslint-config-next: 16.2.1`
- `@vitejs/plugin-react: 4.5.2`
- All `@types/*` packages (`@types/node: 22.19.9`, `@types/react: 19.2.14`, etc.)

**Risk:** `pnpm build` breaks. All TypeScript errors become build errors. The spec's first success criterion ("pnpm build passes") would fail before any collection code is even touched.

**Required fix:** Either pin TypeScript to `^5.8.x` (latest stable 5.x), or provide evidence that TypeScript 6 is compatible with every pinned `@types/*` and ESLint dependency before including it in this RFC. If TS 6 compatibility is desired, it should be its own RFC.

---

### 🟡 M-1: Four `@payloadcms/*` packages are pinned without `^` and are not listed for update

**Spec says:** bump `payload: 3.80.0 → 3.82.0 (and all @payloadcms/* plugins)`

In `package.json`, four packages are pinned at exactly `3.80.0` with no range operator:
- `@payloadcms/db-postgres: 3.80.0`
- `@payloadcms/next: 3.80.0`
- `@payloadcms/richtext-lexical: 3.80.0`
- `@payloadcms/ui: 3.80.0`

These will NOT be upgraded by `pnpm update` unless explicitly listed. The executor will need exact `sed` or manual edits for each. The decision file must enumerate all four explicitly.

## Section 2 — Members Collection

### 🟡 M-2: `cedula` regex allows structurally invalid strings

**Spec says:** `^[0-9.]+$`, minLength 6, maxLength 12

This regex permits `...`, `1.2`, `1..2`, leading and trailing dots. A Colombian cédula formatted with thousand-separators looks like `1.234.567` (7 digits, 2 dots = 9 chars). The regex should enforce the dot-as-separator pattern and disallow consecutive dots:

```
^[0-9]{1,3}(\.[0-9]{3})*$
```

This allows `123456`, `1.234.567`, `12.345.678` and rejects `..123` and `1.2.3`. The min/max constraints alone do not cover this.

**Required fix:** Tighten the regex to `^[0-9]{1,3}(\.[0-9]{3})*$` or accept an all-digits-only format `^[0-9]{6,12}$` (no dots). Pick one and document it in the decision file.

---

### 🔴 H-1: `isActive: false` enforcement is unspecified — inactive lawyers can still authenticate

**Spec says:** Success criterion 6: "isActive: false lawyers cannot access the 'Pool' or any cases."

**The problem:** `isActive` is a data field on the `members` document. It has no effect on authentication or API access by default. A lawyer with `isActive: false` who already has a valid `payload-token` JWT can still make all API calls that their role permits. Payload CMS does not check arbitrary document fields at the auth layer.

**There are two places the spec must specify enforcement, and it specifies neither:**

1. **Login-time gate:** A `beforeOperation: 'login'` hook on `Members` that throws if `isActive: false`. This prevents inactive lawyers from getting new tokens.
2. **Per-request gate:** All access control functions that grant abogado-level access must include `AND isActive: true` in their Where clauses, so existing sessions also fail.

Without both, the Playwright test in criterion 6 would need to test a very specific scenario (existing session, not just failed login). The spec must be explicit about which mechanism is used.

**Required fix:** Spec must specify the `beforeOperation: 'login'` hook that blocks inactive members from authenticating.

---

### 🔴 H-2: Complex abogado read-access query is not specified

**Spec says:** `read: Abogados can read Clientes assigned to their cases.`

The existing `access.owner('id')` pattern returns `{ id: { equals: req.user?.id } }`. This pattern cannot express "read members who are `responsable` in any Case where `abogadoAsignado` equals `req.user.id`."

That query requires a join or subquery via Payload's `where` API that references the `cases` collection. Payload's built-in `Where` object supports relationship field filtering, but the exact query (`{ id: { in: [list of responsable IDs from cases where abogadoAsignado = me] } }`) requires an async pre-computation step — it cannot be expressed as a pure synchronous `Where` object.

**The spec also does not address:** what does an abogado with `role: 'abogado'` look like to the existing `access` utility? The current `access.ts` derives roles from `user.roles` (an array field, present on `Users` collection) via `ExtractRoles`. Members have `role` (singular select). This must be verified at the type level — `ExtractRoles` handles both patterns, but the runtime `processRoles` path must be confirmed.

**Required fix:** The decision file must include the exact async access function code for the abogado read path, not a description.

## Section 3 — Cases Collection

### ⛔ B-2: `caseNumber` format `SUC-YYYYMMDD-HHMM` is not unique

**Spec says:** `caseNumber: text, unique. Format: SUC-YYYYMMDD-HHMM.`

This format has minute-level precision. Two cases created within the same minute (very likely in a multi-user system) will generate identical `caseNumber` values. The database unique constraint will reject the second insert with a 500 error exposed to the client.

**Required fix:** Append a random 4-character alphanumeric suffix:  
`SUC-YYYYMMDD-HHMM-XXXX` (e.g., `SUC-20260408-1430-7K3F`)

Or use a short UUID segment. The uniqueness must be probabilistically guaranteed at the generator level, not just via DB constraint (which is a safety net, not a design strategy).

---

### 🟡 M-3: `responsable` auto-population is not specified

**Spec says:** `responsable: relationship → members, required.`

Who sets `responsable`? If a client creates their own case, `responsable` should be auto-set to `req.user.id` via a `beforeChange` hook. If an admin creates the case, it may be set manually. The spec says nothing about this. If the field is `required` and not auto-populated, the create form in `/admin` will force a manual selection, which may be fine for the MVP — but this assumption must be documented.

---

### 🟡 M-4: `notasInternas` field-level access is mentioned but not specified

**Spec says:** `notasInternas: richText (Lawyer only)`

This implies field-level access control. Payload field access uses `access.read` and `access.update` at the field definition level. The spec does not provide the field access function. The decision file must include it.

## Section 4 — Appointments Collection (Credit Hook)

### ⛔ B-3: Credit hook logic has a semantic error

**Spec says:**
> `afterChange`: If `autorizarCredito` changes to `false` AND `creditoApplied` is `false`, subtract `monto` from `Member.creditoAcumulado`.

This is logically wrong. If `creditoApplied` is `false`, credit was never applied — there is nothing to subtract. The correct reverse-credit condition is:

- `autorizarCredito` changes from `true` to `false` **AND** `creditoApplied` is **`true`**

Then subtract `monto` and set `creditoApplied: false`.

The spec as written would subtract credit every time an admin sets `autorizarCredito` to false on a new, unpaid appointment, driving `creditoAcumulado` negative.

**Required fix:** Correct the condition in the spec to `creditoApplied: true` for the subtraction branch.

---

### ⛔ B-4: `creditoApplied` is never set to `true` — double-application is guaranteed

**Spec says:** add `monto` to `Member.creditoAcumulado` when `autorizarCredito → true AND status = 'realizada'`

The hook description never mentions setting `creditoApplied: true` after the credit is applied. Without this step:

1. Paola marks `autorizarCredito: true`. Hook fires → credit added. `creditoApplied` stays `false`.
2. Any subsequent save of the Appointment document (e.g., updating `status`) fires the `afterChange` hook again.
3. Hook sees `autorizarCredito = true AND status = 'realizada'` → adds credit again.

This will silently inflate `creditoAcumulado` with every edit to the Appointment document.

**Required fix:** The hook must set `creditoApplied: true` via `payload.update` (or via `doc.creditoApplied` in the returned data) immediately after the first successful credit application.

---

### 🔴 H-3: No concurrency protection on credit application

Two simultaneous requests on the same Appointment (e.g., rapid double-click on the admin toggle) will both pass the `creditoApplied: false` guard before either sets it to `true`. This is a TOCTOU race condition.

**Required fix:** The credit application must use a Neon/PostgreSQL atomic operation. Use `payload.db.drizzle.update().set({ creditoAcumulado: sql\`creditoAcumulado + ${monto}\` }).where(...)` within a transaction, OR use an advisory lock. The decision file must specify the exact Drizzle transaction call.

---

### 🔴 H-4: Subtraction can drive `creditoAcumulado` below zero

`Members.creditoAcumulado` has `min: 0`. If the Payload field validator runs on the Member document after the hook subtracts `monto`, and the result is negative, Payload will throw a validation error and the operation will fail silently from the Appointments hook context (hooks don't surface Member validation errors).

**Required fix:** The hook must check `currentMember.creditoAcumulado >= monto` before subtracting. If the guard fails, log an error and skip the subtraction (or alert admin). The decision file must include this guard.

## Section 6 — Access Control Architecture

The existing `access.ts` utility was designed for the `Users` auth collection which uses a `roles: string[]` array. The `Members` collection uses a single `role: select` field. The `ExtractRoles` type handles both patterns (union of `roles` array members and `role` single value), but `processRoles` at runtime uses `user.roles` (array). When a Member logs in as an abogado, `req.user.role` will be `'abogado'` but `req.user.roles` will be `undefined`.

**This means `access.roles('abogado')` will not work as expected for the Members collection.** Custom access functions will need to read `req.user.role` directly, not use the `access.roles()` helper.

The decision file must document this explicitly and provide raw access function implementations, not calls to `access.roles()`.

## Recommended Path Forward

1. **Drop TypeScript 6.0.2 from this RFC.** Pin `typescript: ^5.8.x` instead. Open a separate RFC for the TS 6 upgrade after compatibility is confirmed.
2. **Fix B-2:** Change `caseNumber` format to `SUC-YYYYMMDD-HHMM-XXXX` with a random 4-char suffix.
3. **Fix B-3 + B-4:** Rewrite the credit hook spec with the correct conditions and the `creditoApplied: true` setter.
4. **Specify H-1:** Add a `beforeOperation: 'login'` hook that rejects `isActive: false` members.
5. **Specify H-2:** The decision file must include raw access functions for Members, not `access.roles()` calls.
6. Enumerate all 4 pinned `@payloadcms/*` packages explicitly in the decision file task.
