## 1. Canonical Decisions

### 1.1 Final slugs and status vocabulary

Use these exact collection slugs everywhere:

- `documents`
- `document-checklists`
- `notary-process`
- `payments`

Use `approved`, not `verified`, for legal document/checklist approval.

Checklist statuses:

- `pending`
- `uploaded`
- `approved`
- `rejected`

Document statuses:

- `uploaded`
- `approved`
- `rejected`
- `replaced`

Before implementation, scan the repo for conflicting names:

```bash
rg "document-checklist|document-checklists|notary-process|notaryProcess|verified|approved" .agents docs src tests
```

Resolve only RFC-003.3-relevant references. Do not rewrite unrelated historical archives unless needed by active code/tests.

### 1.2 Role model

- **client:** `members` auth user with `role: 'cliente'`.
- **lawyer:** `members` auth user with `role: 'abogado'` and `isActive !== false`.
- **staff/admin:** internal Payload `users` accounts. Use least-privilege access.
- **super-admin:** future break-glass role. For this RFC, any override/correction path must require `overrideReason`; full audit logs are deferred to `GOV-001`.

### 1.3 Documents access model

- Clients can read `visibility: 'client'` documents for their own cases.
- Assigned active lawyers can read documents for assigned cases.
- Unassigned lawyers cannot read unrelated case documents.
- Staff/admin access must be least-privilege.
- Clients can upload documents for their own cases.
- Clients do not mutate reviewed documents in place; corrections create new document versions.
- Admin/super-admin delete only; prefer versioning over deletion.

### 1.4 Checklist model

Checklist items can be:

- `source: 'system'`: generated by workflow hooks.
- `source: 'manual'`: added by the assigned lawyer or authorized staff/admin.

Manual items must support `required: true | false`.

- Required manual items block the configured phase until approved.
- Optional manual items appear in the checklist but do not block advancement.

### 1.5 Payment model

Payments are provider-agnostic business records, not Wompi records.

Manual/offline payments are first-class:

- bank transfer
- cash
- QR
- Nequi
- other

Wompi-specific automation is deferred to DEC-005. RFC-003.3 may store `paymentProvider: 'wompi'` and `providerTransactionId`, but must not implement Wompi checkout/webhooks unless Teo expands scope.

### TASK 5 — Implement checklist generation engine

**Files:**

- Create: `src/payload/collections/succession/DocumentChecklist/utils/generateChecklistItems.ts`
- Create: `src/payload/collections/succession/Cases/hooks/generateInitialChecklist.ts`
- Create: `src/payload/collections/succession/Heirs/hooks/generateHeirChecklistItems.ts`
- Create: `src/payload/collections/succession/Assets/hooks/generateAssetChecklistItems.ts`
- Modify: `src/payload/collections/succession/Cases/index.ts`
- Modify: `src/payload/collections/succession/Heirs/index.ts`
- Modify: `src/payload/collections/succession/Assets/index.ts`
- Create: `tests/int/checklistGeneration.int.spec.ts`

**Generation contract:**

1. On case creation, generate exactly one required Poder item:
   - `name: "Poder de Representación Notarial"`
   - `category: 'poder'`
   - `documentType: 'poder'`
   - `required: true`
   - `requiredForPhase: 4`
   - `source: 'system'`
2. When a case enters phase 2, generate:
   - causante defunción/cédula items
   - heir items for all existing heirs
   - asset items for all existing assets
3. When a new heir is created after phase 2, generate that heir's items.
4. When a new asset is created after phase 2, generate that asset's items.
5. Generation must be idempotent.

**Minimum required item matrix:**

- causante:
  - registro civil de defunción
  - copia de cédula
- normal heir:
  - registro civil de nacimiento
  - copia de cédula
- deceased heir:
  - registro civil de defunción
- representative heir:
  - registro civil proving lineage
  - copia de cédula
- asset `inmueble`:
  - certificado de tradición
  - escritura pública de tradición
- asset `vehiculo`:
  - tarjeta de propiedad
  - certificado de tradición del vehículo
- asset financial/cuenta:
  - certificado bancario or extract

If existing asset enum values differ, map from current values in `Assets/index.ts` and document the mapping in code comments.

**Tests:**

- Poder generated at case creation
- phase 2 generation creates expected causante/heir/asset items
- running generation twice creates no duplicates
- adding heir after phase 2 creates only that heir's items
- adding asset after phase 2 creates only that asset's items

### TASK 9 — Implement phase validation and case advancement endpoint

**Files:**

- Create: `src/payload/collections/succession/Cases/utils/validatePhaseTransition.ts`
- Create: `src/payload/collections/succession/Cases/endpoints/advancePhase.ts`

**Endpoint:**

- `POST /api/cases/:id/advance-phase`
- collection endpoint on `cases`: `path: '/:id/advance-phase'`
- use `req.routeParams.id`

**Validation rules:**

- only assigned active lawyer or authorized staff/admin can advance
- normally only `currentPhase -> currentPhase + 1`
- direct `currentPhase` updates must use same validation
- Phase 3 -> 4:
  - all required relevant checklist items approved
  - exactly one required Poder item approved
  - `receivedPhysically` does not satisfy this gate
- Phase 4 -> 5:
  - `notary-process.respuestaNotario.status === 'aprobado'`
- Phase 5 -> 6:
  - `notary-process.edictos.comprobantesEntregados === true`
- Phase 6 -> 7:
  - DIAN and UGPP status are `aprobado`
- Phase 7 -> 8:
  - `firma.escrituraPublica` exists
  - if `case.tier === 'estandar'`, do not move to phase 8; mark case completed per DEC-004
  - if `premium` or `elite`, phase 8 is allowed
- Phase 8 -> completed:
  - `registro.status === 'registrado'`

**Errors:** return explicit Spanish errors with 4xx status. Example:

```text
El Poder de Representación Notarial debe estar cargado y aprobado antes de radicar en notaría.
```

**Tests:**

- unauthorized client cannot advance
- unassigned lawyer cannot advance
- phase 3 -> 4 blocked without approved Poder
- `receivedPhysically` alone still blocks
- phase 4 -> 5 blocked without notary approval
- estándar tier completes at signature instead of moving to phase 8
- premium tier can move to phase 8 when signature doc exists
