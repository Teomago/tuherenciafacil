# RFC-003.3: Workflow Engine & Legal Operations

## Objective
Complete the workflow-domain backend required to operate succession cases safely and consistently before major UI funnel expansion. This establishes the data models and constraints for document lifecycles, checklist tracking, notary processes, phase gates, and provider-agnostic payment foundations.

## Scope Lock
- RFC-003.3 is backend/admin workflow infrastructure only.
- Do not implement app shell or component-system work (`RFC-004`).
- Do not implement onboarding or pre-payment screen orchestration (`RFC-005`).
- Do not implement broad client dashboard/workspace UI beyond what is required for admin/backend verification.
- If execution requires behavior outside this scope and the behavior is not explicitly in the decision file, stop and escalate.

## Role Model
- **client**: product user in `members` with `role: 'cliente'`.
- **lawyer**: product user in `members` with `role: 'abogado'` and `isActive !== false`.
- **staff/admin**: internal Payload `users` accounts. Use least-privilege access for sensitive legal/payment data.
- **super-admin**: future break-glass role allowed to override sensitive records with required `overrideReason`. Full audit logging is tracked separately in `GOV-001`.

## Field-by-Field Implementation Details

### 1. Documents Collection
- Dedicated Payload upload collection, not a relationship to `media`.
- **`slug`**: `documents`.
- **Upload config**: collection-level `upload` configuration, R2-backed through `@payloadcms/storage-s3`.
- **Allowed file types**: PDF, JPEG, PNG.
- **`case`**: relationship to `cases`, required.
- **`type`**: select (`registro_civil`, `cedula`, `poder`, `certificado_tradicion`, `escritura`, `notarial`, `otro`), required.
- **`status`**: select (`uploaded`, `approved`, `rejected`, `replaced`), default `uploaded`.
- **`uploadedBy`**: relationship to `members` or `users`, required.
- **`reviewedBy`**: relationship to `members` or `users`.
- **`reviewedAt`**: date.
- **`rejectionReason`**: textarea, required when status becomes `rejected`.
- **`visibility`**: select (`client`, `internal`), default `client`.
- **`version`**: number, default `1`.
- **`previousVersion`**: relationship to `documents`.
- **File delivery**: do not rely only on generic CRUD URLs. Add a controlled signed-URL path for document file access, with permission checks and short TTL.

### 2. DocumentChecklist Collection
- **`slug`**: `document-checklists`.
- **`case`**: relationship to `cases`, required.
- **`name`**: text, required.
- **`description`**: text.
- **`guiaDePaola`**: text.
- **`documentType`**: select matching `documents.type`, required.
- **`category`**: select (`causante`, `heredero`, `bien`, `poder`, `legal`, `notarial`, `custom`), required.
- **`heir`**: relationship to `heirs`, optional.
- **`asset`**: relationship to `assets`, optional.
- **`source`**: select (`system`, `manual`), required, default `system`.
- **`required`**: checkbox, default `true`. Manual/additional documents may be required or optional.
- **`requiredForPhase`**: number, optional. If present and `required === true`, this item blocks advancement into that phase until approved.
- **`requestedBy`**: relationship to `members` or `users`, optional. Required for manual items.
- **`requestedAt`**: date, optional. Required for manual items.
- **`status`**: select (`pending`, `uploaded`, `approved`, `rejected`), default `pending`.
- **`receivedPhysically`**: checkbox, default `false`. Does not satisfy legal gates by itself.
- **`receivedPhysicallyAt`**: date.
- **`document`**: relationship to `documents`, optional.
- **`reviewNote`**: textarea.
- **`reviewedBy`**: relationship to `members` or `users`.
- **`reviewedAt`**: date.
- **Manual/additional document rule**: the assigned lawyer can add case-specific checklist items and decide whether each one is required. Required manual items must block the relevant phase; optional manual items do not block advancement.

### 3. NotaryProcess Collection
- Implement the complete backend schema now, even if UI is deferred.
- **`slug`**: `notary-process`.
- **`case`**: relationship to `cases`, required, unique.
- Include structured groups for:
  - `radicacion`
  - `respuestaNotario`
  - `edictos`
  - `dian`
  - `ugpp`
  - `firma`
  - `costosNotariales`
  - `registro`
- These fields are required so phase gates 4 through 8 can be validated without a future schema rewrite.
- Client access to raw `notary-process` records is out of scope unless field-level redaction is explicitly added. Client-facing progress belongs to later UI/timeline work.

### 4. Payments Collection
- Provider-agnostic payment ledger. Do not make Wompi the center of the data model.
- **`slug`**: `payments`.
- **`case`**: relationship to `cases`, optional, because some payments may exist before a case.
- **`member`**: relationship to `members`, required when `case` is empty.
- **`tipo`**: select aligned with DEC-004 (`consulta`, `servicio`, `creditoConsulta`, `investigacion`, `gasto_tercero`, `cuota`, `ajuste`, `otro`).
- **`amount`**: number, required, COP.
- **`currency`**: text, default `COP`.
- **`paymentProvider`**: select (`manual`, `wompi`, `other`), default `manual`.
- **`method`**: select (`bank_transfer`, `cash`, `qr`, `nequi`, `pse`, `card`, `other`).
- **`status`**: select (`requested`, `pending_confirmation`, `approved`, `failed`, `cancelled`, `refunded`), default `requested`.
- **`paymentReference`**: text.
- **`providerTransactionId`**: text. Used by Wompi or future providers.
- **`visibleToClient`**: checkbox, default `true`.
- **`registeredBy`**: relationship to `members` or `users`, required for manual records.
- **`approvedBy`**: relationship to `members` or `users`.
- **`approvedAt`**: date.
- **`overrideReason`**: textarea, required for super-admin overrides or direct corrections.
- **Manual payment rule**: staff/admin/lawyer can register real-world payments such as bank transfer, QR, Nequi, or cash according to access rules. This lets the workflow advance without a payment provider integration.
- **Wompi boundary**: RFC-003.3 creates the provider-agnostic foundation only. Wompi webhooks, checksum validation, checkout redirects, retries, and automated reconciliation remain in the future DEC-005 implementation unless explicitly added to the decision file.

## API Endpoint Specifications
- Payload CRUD is not sufficient for all RFC-003.3 behavior.
- Add a controlled case advancement endpoint:
  - `POST /api/cases/:id/advance-phase`
  - Implement as a Payload collection endpoint on `cases` using `path: '/:id/advance-phase'`.
  - Payload supports dynamic `:id` route parameters via `req.routeParams.id`.
  - The endpoint validates permissions, computes the next phase, applies phase gates, returns Spanish error messages, and updates `currentPhase` only when legal gates pass.
- Add a controlled document signed URL path:
  - `GET /api/documents/:id/url`
  - Implement as a Payload collection endpoint on `documents` using `path: '/:id/url'`.
  - The endpoint checks document read access and returns a short-lived signed URL instead of exposing private files directly.

## Access Control Rules
- **Documents:**
  - `read`: Clients can read `visibility: 'client'` documents linked to their own cases. Assigned active lawyers can read documents for assigned cases. Staff/admin access should use least privilege. No unassigned lawyer can read unrelated case documents.
  - `create`: Clients can upload documents for their own cases; assigned active lawyers and staff/admin can upload documents for authorized cases.
  - `update`: Clients do not update documents in place; corrections create a new version. Assigned active lawyers and staff/admin can review/approve/reject authorized case documents.
  - `delete`: Admin/super-admin only. Prefer versioning over deletion.
- **DocumentChecklist:**
  - `read`: Clients can read checklists for their cases.
  - `create`: System hooks, assigned active lawyer, or authorized staff/admin.
  - `update`: Assigned active lawyer or authorized staff/admin. Clients do not update checklist records directly.
  - `delete`: Admin/super-admin only.
- **NotaryProcess:**
  - `read`: Assigned active lawyer and authorized staff/admin only. Raw records are not client-readable in RFC-003.3.
  - `create`/`update`: Assigned active lawyer and authorized staff/admin.
  - `delete`: Admin/super-admin only.
- **Payments:**
  - `read`: Clients can read visible payments linked to their own case/member. Assigned active lawyers can read payments for assigned cases. Staff/admin access should use least privilege.
  - `create`: Assigned active lawyer or staff/admin for manual records; system hooks for future providers.
  - `update`: Prefer append-only corrections. Admin/super-admin may correct records with `overrideReason`; future audit logging tracked in `GOV-001`.
  - `delete`: Admin/super-admin only.

## Hooks and Triggers
- **Cases `beforeChange` Hook (`validatePhaseTransition`):**
  - Evaluates updates to `currentPhase`.
  - Direct `currentPhase` updates must not bypass the same validation used by `POST /api/cases/:id/advance-phase`.
- **Checklist generation:**
  - On case creation: generate the required Poder item immediately.
  - When a case enters phase 2: generate causante, heir, and asset checklist items.
  - When a new heir is added after phase 2: generate that heir's required items.
  - When a new asset is added after phase 2: generate that asset's required items.
  - Generation must be idempotent and must not duplicate existing checklist items.
- **Document-checklist sync:**
  - On document upload linked to a checklist item: mark checklist item `uploaded`.
  - On lawyer approval: set both document and checklist item to `approved`, with reviewer and timestamp.
  - On rejection: set both records to `rejected`, require a rejection/review note, and allow a replacement upload.
- **Phase gates:**
  - Phase 3 -> 4 requires all required relevant checklist items approved and exactly one approved Poder item. `receivedPhysically` does not satisfy this gate.
  - Later phase gates use `notary-process` structured fields, including notary response, edicts, DIAN/UGPP, signature, and registration.
  - Phase 7 -> 8 is tier-dependent per DEC-004.
