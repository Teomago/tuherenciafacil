## Executive Summary
- **Integrity Guards:** Constrain `responsable` to Clients and `abogadoAsignado` to active Lawyers.
- **Intake Validation:** Enforce 1+ heirs for intake and intestate-only guards.
- **Lineage Triggers:** Automatically add Birth Certificates to the checklist for heirs in representation.
- **Scheduling MVP:** Add Saturday morning availability slots to Appointments.
- **Phase Gate Polish:** Add "Power of Attorney" (Poder) validation to the `advance-phase` logic.

---

## Problem Statement
The raw implementation of RFC-003.3 provides the core collections but lacks the validation layer and admin UX polish required for safe legal operations. In `/admin`, it is currently possible to assign invalid roles or bypass critical legal gates (like the Power of Attorney check), which creates high operational risk for Paola.

---

## Goals
- **Role Enforcement:** Strict filtering in `Cases` relationships.
- **Intake Protection:** Validate for minimum heirs and intestate eligibility before Case creation.
- **Lineage Triggers:** Automatically add Birth Certificates to the checklist for heirs in representation.
- **Scoped Pickers:** Ensure all case-related relationships in the admin panel are filtered to the current case context.

---

## Requirements by Collection

### 1. Cases (Relationship & Phase Integrity)
- **`responsable`:** Must show only `Members` with `role: 'cliente'`.
- **`abogadoAsignado`:** Must show only `Members` with `role: 'abogado'` and `isActive: true`.
- **Phase Gate (Fase 3 → 4):** Modify the `advance-phase` logic (hook or endpoint) to verify that a `document` of type `poder` exists in the `DocumentChecklist` and is `approved`.

### 2. Case Intakes (Guardrails)
- **Heir Guard:** Validation prevents submission if `herederosEstimados < 1`. (Supports single-heir successions).
- **Eligibility:** Hard-block conversion to `Case` if `tieneTestamento` is true.
- **Data Completeness:** `Submitted` status requires `causante.fechaFallecimiento`.

### 3. Appointments (Saturday Scheduling)
- **Schema:** Add `fechaAgendada` (Date field) and `status: 'realizada'`.
- **Availability:** Data model must support Saturday morning slots (8:00 AM - 12:00 PM).
- **Credit Logic:** `autorizarCredito` updates `Member.creditoAcumulado` (DEC-009).

### 4. Heirs (Representation Flow)
- **Lineage Verification:** If `esRepresentante` is true, the system must automatically add "Registro Civil de Nacimiento (Representante)" to the `DocumentChecklist`.
- **Validation:** `herederoOriginal` must link to a deceased person in the same case.

### 5. Assets (Investigation)
- **Evidence Model:** `investigacion.resultado` links to `Documents` collection.
- **Cost Policy:** If `tier: 'estandar'`, investigation triggers a $150k pending cost.

### 6. Document Checklists & Documents
- **Scoping:** The `document` picker in `DocumentChecklist` must be filtered by `case` ID to prevent data leakage between families.
- **Taxonomy:** Admin labels should follow `[Category] - [Entity Name] - [Doc Name]`.
- **Rejection Reasons:** Add `rejectionReason` (ilegible, vencido, incompleto, no_corresponde) to `DocumentChecklist`.

### 7. Notary Process (Lawyer UX)
- **Layout:** Group fields by Phase (Fase 4-8) using UI groups.
- **Scoping:** Filter all document relationships strictly to the current `case`.

### 8. Payments (Manual Ledger)
- **Visibility:** `visibleParaCliente` toggle to distinguish between internal costs and client-facing billing.

- Chat/Messaging functionality.
- Wompi Payment automation/webhooks (DEC-005).
