## Section 1 — Cases (Role Enforcement & Phase Integrity)

### HIGH H-1: `responsable` and `abogadoAsignado` pickers have no role filtering or API enforcement

**Spec says:** `responsable` must show only `Members` with `role: 'cliente'`; `abogadoAsignado` must show only `Members` with `role: 'abogado'` and `isActive: true`.

**Current code:** Both fields in `Cases/index.ts` (lines 110-118) are plain `relationship` fields to `members` with zero `filterOptions` and zero `beforeChange` validation. Any `members` record can be assigned to either field via the admin UI or API.

**Problem:** `filterOptions` only protects the admin panel UI picker. A direct API call (`payload.update`) can set `responsable` to a lawyer or `abogadoAsignado` to a client without any validation.

**Required fix for decision file:**

1. Add `filterOptions` on `responsable`:
   ```typescript
   filterOptions: () => ({ role: { equals: 'cliente' } })
   ```

2. Add `filterOptions` on `abogadoAsignado`:
   ```typescript
   filterOptions: () => ({
     role: { equals: 'abogado' },
     isActive: { not_equals: false },
   })
   ```

3. Add a new `beforeChange` hook (`validateRoleAssignments.ts`) that enforces:
   - If `responsable` changed → fetched member must have `role === 'cliente'`
   - If `abogadoAsignado` changed → fetched member must have `role === 'abogado'` and `isActive !== false`
   - Errors returned in Spanish with 422 status

## Section 2 — Case Intakes (Guardrails)

### HIGH H-3: `tieneTestamento` hard-block location and existing data handling

**Spec says:** "Hard-block conversion to `Case` if `tieneTestamento` is true."

**Current code:** `CaseIntake/index.ts` has `tieneTestamento` as a checkbox with `admin: { readOnly: true }`. The `beforeChange` hook validates `tierElegido` and `acuerdoHerederos` on submission but does not check `tieneTestamento`. The `convert-intake` endpoint (from RFC-003.2) is where case creation logic lives.

**Problem:** Two possible locations:
1. `CaseIntake.beforeChange` hook — would prevent saving an intake with `tieneTestamento: true`, which is too aggressive (the client should see an informative message in the UI, not a backend crash)
2. `convert-intake` endpoint — validates only at conversion time, which is the correct legal gate

**Required fix for decision file:** Place the testamento block at the **`convert-intake` endpoint level** only. The intake should remain saveable as draft with `tieneTestamento: true`. The conversion endpoint returns a 422 with a clear Spanish message:
```text
Las sucesiones testadas no son atendidas por tuHerenciaFácil. Solo manejamos sucesiones intestadas.
```

No existing-record exemption is needed — the database only contains test data and can be reset safely.

## Section 3 — Heirs (Representation Flow)

**Problems:**

2. Re-implementing without acknowledging the existing code will create duplicate checklist items or conflicting hooks.

**Required fix for decision file:** Compare the existing implementation against the RFC-003.4 spec's desired output. If the current implementation already covers the intent (automatic checklist item for representation), no code change is needed. If the `documentType` or item name needs refinement, frame it as a targeted update to `generateChecklistItems.ts`, not a new feature.

### HIGH H-4: `herederoOriginal` has no case-scope filter or `esFallecido` validation

**Spec says:** "`herederoOriginal` must link to a deceased person in the same case."

**Current code:** `Heirs/index.ts` (lines 77-84) has `herederoOriginal` as a plain relationship to `heirs` with only an `admin.condition` that shows/hides the field. There is:
- No `filterOptions` to scope to same-case heirs
- No validation that the linked heir has `esFallecido === true`
- No validation that the linked heir belongs to the same case

**Risk:** Cross-case data contamination — a representative heir in Case A could link to a deceased heir in Case B. Additionally, a living heir could be incorrectly selected as the original, which is legally invalid under Colombian representation rights (*derecho de representación*).

**Required fix for decision file:**

1. Add `filterOptions` on `herederoOriginal`:
   ```typescript
   filterOptions: ({ data }) => {
     if (!data?.case) return false
     const caseId = typeof data.case === 'object' ? data.case.id : data.case
     return {
       case: { equals: caseId },
       esFallecido: { equals: true },
     }
   }
   ```

2. Add `beforeChange` hook on Heirs:
   - If `esRepresentante === true` and `herederoOriginal` is empty → error 422: `"Un heredero representante debe tener vinculado al heredero original fallecido."`
   - If `esRepresentante === true` and `herederoOriginal` is set → fetch the original, validate:
     - `esFallecido === true` → else error: `"El heredero original debe estar marcado como fallecido."`
     - Same `case` → else error: `"El heredero original debe pertenecer al mismo caso."`

## Section 4 — Appointments (Availability Model)

### HIGH H-5: Saturday scheduling is underspecified and requires a new `availability-slots` collection

**Spec says:** "Add `fechaAgendada` and `status: 'realizada'`. Data model must support Saturday morning slots (8:00 AM - 12:00 PM)."

**Problem:** A date field alone does not enforce time windows. The spec does not describe:
- How availability is modeled
- How clients book into available slots
- Whether double-booking is prevented
- Cancellation/rescheduling policy
- Payment requirement for appointments

**Required fix for decision file:** Create a new `availability-slots` collection and extend `Appointments` with booking mechanics:

**`availability-slots` collection:**
- `slug: 'availability-slots'`
- `lawyer`: relationship to `members` (required)
- `date`: date (required)
- `startTime`: text or date (required) — e.g., "08:00"
- `endTime`: text or date (required) — e.g., "10:00"
- `appointmentDuration`: number in minutes (required) — e.g., 60 or 120
- `maxAppointments`: number (required) — calculated or set by lawyer (how many appointments fit in the window, given duration)
- `status`: `open | full | cancelled`, default `open`
- Access: Lawyer/admin can create/update/delete; clients can read (to see available times)

**`appointments` schema additions:**
- `fechaAgendada`: date (required)
- `availabilitySlot`: relationship to `availability-slots`
- `status`: add `'realizada'` to existing options
- `rescheduleDeadline`: computed or enforced — at least 24 hours before `fechaAgendada`

**Booking behavior:**
- Paola creates availability slots with any day/time she wants (not hard-coded to Saturdays)
- She controls duration per slot (warning if > 4 hours: "Reuniones de más de 4 horas no son recomendadas")
- Client selects an open slot → appointment is immediately confirmed (no approval flow)
- Slot becomes `full` when `maxAppointments` reached
- Appointments are paid — payment is required to confirm the booking (ties into `payments` collection)
- Client cannot cancel but can reschedule with 24+ hours notice. Under 24 hours → rescheduling blocked, reminder email sent 1 hour before
- Paola can reschedule at any time (sends email notification to client)
- Appointments work for both authenticated members AND guests without accounts

**Deferred to later RFCs:**
- Email reminders (notification system — track in backlog)
- Client-facing booking UI (RFC-004/005)

## Section 5 — Document Checklists & Documents (Case Scoping)

### HIGH H-6: `filterOptions` for case-scoped pickers is technically underspecified

**Spec says:** "The `document` picker in `DocumentChecklist` must be filtered by `case` ID to prevent data leakage between families."

**Problem:** The spec doesn't specify the exact `filterOptions` callback, which field path to use, or whether API-level enforcement is also needed.

**Required fix for decision file:** Add `filterOptions` using `data.case` (top-level field) on the `document` picker in `DocumentChecklist`:

```typescript
filterOptions: ({ data }) => {
  if (!data?.case) return false
  const caseId = typeof data.case === 'object' ? data.case.id : data.case
  return { case: { equals: caseId } }
}
```

**API-level enforcement:** Not needed for the `document` picker specifically — the existing access control on the `documents` collection already enforces case-scoped reads (client sees only own-case docs, lawyer sees only assigned-case docs). `filterOptions` here is a UX improvement, not a security boundary.

Same pattern applies to document pickers in `NotaryProcess` (filter by `data.case`).
